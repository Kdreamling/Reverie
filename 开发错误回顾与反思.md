# 开发错误回顾与反思

> 日期：2026-03-10
> 项目：Reverie 前端

---

## 错误一：create-vite 初始化被取消

### 现象
运行 `npm create vite@latest . -- --template react-ts` 时，因目录非空出现交互提示，尝试 `echo y |` 管道输入仍被取消。

### 根因
`create-vite` 在非空目录下会弹出交互式确认，管道输入无法触达该提示（它读的是 TTY 而非 stdin）。

### 处理方式
放弃调 CLI，手动逐一创建 `package.json`、`vite.config.ts`、`tsconfig*.json`、`index.html`、`src/main.tsx` 等所有模板文件。

### 反思
遇到交互式 CLI 无法自动化时，应直接手写文件，不要反复尝试管道/重定向，浪费时间。

---

## 错误二：字符串中的弯引号导致 TypeScript 报错

### 现象
`ChatPage.tsx` 构建时出现大量 `TS1005: ',' expected` 错误，指向假消息数据里的几行字符串。

```
src/pages/ChatPage.tsx(37,75): error TS1005: ',' expected.
```

### 根因
写假消息内容时用了 `'`（Unicode 弯单引号 `\u2019`），例如 `"I'm"` 被写成了 `'I'm'`，导致 TS 把弯引号里的内容解析为多个 token。

```ts
// 错误：弯引号结束了字符串
content: 'I'm curious...'

// 正确：改用双引号包裹含单引号的字符串
content: "I'm curious..."
```

### 处理方式
将包含撇号的字符串由单引号改为双引号包裹。

### 反思
写含缩写词（I'm、What's）的英文字符串时，优先用双引号或模板字符串，避免引号冲突。在 AI 辅助写代码时尤其要注意，因为 AI 可能输出 Unicode 弯引号。

## 错误三：client.ts 缺少 patch / delete 方法

### 现象
实现 session CRUD 时发现 `client` 对象只有 `get` 和 `post`，需要手动补充。

### 根因
最初只考虑了登录（POST）和查询（GET）场景，没有预见到后续的修改和删除需求。

### 反思
API 封装层应在设计时就包含完整的 HTTP 方法（GET/POST/PATCH/PUT/DELETE），避免后续补丁式扩展。也应在初版就处理好 204 No Content 的特殊情况。

---

## 错误四：401 处理的循环依赖问题

### 现象
想在 `client.ts` 里直接调 `authStore.logout()` 处理 401，但：
- `client.ts` → 不导入任何模块
- `authStore.ts` → 导入 `api/auth.ts` → 导入 `client.ts`

形成循环依赖，不能直接引用。

### 处理方式
在 `client.ts` 中改用 DOM 自定义事件：

```ts
window.dispatchEvent(new Event('auth:unauthorized'))
```

在 `App.tsx`（已导入 authStore）监听事件并调 `logout()`。

### 反思
模块间存在循环依赖时，**事件总线**（DOM CustomEvent 或 mitt）是解耦的标准方案。这个模式值得记住：底层模块广播事件，上层模块订阅并处理。

## 错误五：spaFallback 自定义插件拦截了 Vite 内部路径

### 现象
实现 SPA 路由 fallback 时写了一个 Vite 插件，拦截所有 `/chat/*` 请求并重写到 `index.html`。结果 `/@vite/client`、`/@react-refresh` 等 Vite 内部路径也被重写，开发服务器报 500 错误。

### 根因
条件判断 `!url.includes('.')` 无法排除以 `@` 开头的 Vite 内部路径（`/@vite/client` 不含点号）。

```ts
// 原有条件：不含点号就重写 → 错误地重写了 /@vite/client
!url.includes('.')
```

### 处理方式
直接删掉自定义插件，改用 Vite 内置的 `appType: 'spa'`：

```ts
export default defineConfig({
  appType: 'spa',  // 内置 SPA fallback，自动排除内部路径
  ...
})
```

### 反思
**优先用框架内置功能，不要重复造轮子。** Vite 的 `appType: 'spa'` 本身就是为此设计的，且经过充分测试。自己写中间件很容易漏掉边缘情况（Vite 内部路径、WebSocket、`__vite_ping` 等）。

## 错误六：historyApiFallback 不在 Vite 6 的类型定义中

### 现象
尝试在 `vite.config.ts` 里加 `server: { historyApiFallback: true }`，TypeScript 报 `Property 'historyApiFallback' does not exist on type 'ServerOptions'`。

### 根因
Vite 6 移除了 `server.historyApiFallback`（或者从未正式支持过），该选项属于旧版的非官方用法。

### 处理方式
先改为自定义 Plugin（但引入了错误五），最终用 `appType: 'spa'` 解决。

### 反思
使用某个配置项前应先查文档确认它是否存在于当前大版本中，不要凭印象或旧文档操作。Vite 6 是重大版本更新，API 有变化。

## 错误七：Vite Plugin 里访问 req.url 的 TypeScript 类型问题

### 现象
写 `configureServer` 中间件时，`req` 参数的类型是 `IncomingMessage`（Vite 内部定义），TypeScript 提示 `Property 'url' does not exist on type 'IncomingMessage'`。

### 根因
Vite 的 `Connect.IncomingMessage` 类型定义不完整，虽然运行时有 `url` 属性，但 TS 类型不包含它。

### 处理方式
用类型断言绕过：

```ts
const r = req as { url?: string }
```

### 反思
第三方库类型不完整时，用最小范围的类型断言（断言为具体的 shape 而不是 `any`）是合适的临时方案，同时留下注释说明原因。

## 错误八：messages.map is not a function（API 响应格式不符预期）

### 现象
进入聊天页后 Console 报错 `messages.map is not a function`，定位在 `ChatPage.tsx:287`。

### 根因
`fetchMessagesAPI` 返回类型标注为 `ChatMessage[]`，但后端实际返回的可能是 `{ data: [...] }` 包裹格式，导致 `messages` state 被赋值为一个对象而不是数组，调用 `.map()` 报错。

### 处理方式
在 `loadMessages` 中加兼容逻辑 + console.log 打印原始响应：

```ts
const raw = await fetchMessagesAPI(sessionId)
console.log('[chatStore] loadMessages raw response:', raw)
const list = Array.isArray(raw)
  ? raw
  : Array.isArray((raw as { data?: unknown }).data)
    ? (raw as { data: ChatMessage[] }).data
    : []
set({ messages: list })
```

同时在 ChatPage 渲染时加 `Array.isArray(messages)` 防御检查。

## 错误九：AI 流式工具调用参数取值不一致

> 日期：2026-03-14

### 现象
AI 调用 `save_memory` 工具后，前端 MemoryOpsBlock 显示内容为空白，但 Supabase 里确实存了记忆。

### 根因
`save_memory` 工具定义的参数名是 `content`，但 AI 有时会用 `text` 字段传内容。后端取值只用 `args.get("content")`，当 AI 用 `text` 字段时就取到空字符串。

### 修复
```python
content = args.get("content", "") or args.get("text", "")
```
同时前端加兜底显示：`op.content || '（内容为空）'`，以及 `logger.info(f"[reverie] tool call: {fn_name} args={args}")` 方便排查。

### 反思
工具参数名要在 tool definition 和执行代码中严格对齐。当 AI 模型"自由发挥"参数名时，后端应做多字段兜底。

## 错误十：流式输出被工具调用截断

> 日期：2026-03-14

### 现象
AI 在回复过程中尝试调用 `save_memory`，但流式 SSE 没有 tools，导致 AI 无法调用，回复直接截断。

### 根因
旧架构：先做一次非流式 tool phase（让 AI 搜索记忆），执行完后再开流式输出。但 AI 想在流式阶段保存记忆时没有 tools 可用。

### 修复
重写 `_reverie_stream()`，在流式请求中携带 `tools` 参数，实时检测 `delta.tool_calls`，收集 function name + arguments，`finish_reason == "tool_calls"` 时执行工具并重新发起流式请求，最多循环 3 轮。移除 `_reverie_stream_with_pre`。

### 反思
"先 tool 后 stream"的两段式设计存在根本局限，无法支持 AI 在生成过程中动态决策。流式 tool call 循环是 OpenAI function calling 的标准模式，应直接采用。

## 错误十一：memory_ops 刷新后丢失

> 日期：2026-03-14

### 现象
AI 执行了记忆操作（绿色 MemoryOpsBlock 显示正常），服务器重启或页面刷新后绿色框消失。

### 根因
`pendingMemoryOps` 只存在于前端内存，`done` 事件时附加到消息对象，但 `_reverie_store` 存入数据库时没有包含这些数据，`loadMessages` 重新加载时自然丢失。

### 修复
1. 后端 `_reverie_stream` 收集 `collected_memory_ops`，传给 `_reverie_store`
2. `_reverie_store` 将 memory_ops 序列化为 JSON 存入 `conversations.memory_ops`（JSONB 字段）
3. `sessions.py` 的 messages 查询加上 `memory_ops` 字段
4. 前端 `chatStore.loadMessages` 解析 `memory_ops` JSON，映射为 `MemoryOperation[]`

### 反思
任何需要在历史消息中持久显示的 UI 数据，都必须和消息一起存入数据库。不能依赖运行时内存状态。

## 错误十二：SSH 命令中的 shell 引号陷阱

> 日期：2026-03-14

### 现象
在本地终端运行 SSH 命令时，包含单引号的命令卡在 `dquote>` 提示符，无法执行。

### 根因
shell 对引号嵌套解析失败。外层双引号内包含单引号，或反引号嵌套时，shell 会进入引号等待状态。

### 处理方式
Ctrl+C 退出，直接 SSH 登录服务器后在远端执行命令，避免引号嵌套问题。

### 反思
复杂的 SSH 内嵌命令容易触发引号解析问题。对于多步骤操作，优先 SSH 登录后手动执行，而不是拼接长命令字符串。

### 反思
- **API 类型标注要与后端实际行为对齐**，不能只看接口文档，要用 `console.log` 验证真实响应。
- 关键 state（特别是 `[]` 数组）在赋值时应做类型验证，提前 fallback 到空数组，避免组件层崩溃。
- 更根本的解法是用 Zod 等 schema 验证库在 API 层做运行时类型校验。

## 综合反思

### 做得好的

1. **关注点分离清晰**：api 层、store 层、页面层职责单一，没有混用。
2. **循环依赖处理得当**：用 DOM 事件解耦 client.ts 和 authStore.ts。
3. **遇到障碍及时调整**：create-vite 失败立即手写文件，没有在错误路径上纠缠。
4. **构建验证**：每个阶段改完都跑 `npm run build` 确认没有类型错误。

### 需要改进的

1. **先查文档再写配置**：`historyApiFallback` 和 `appType: 'spa'` 的折腾完全可以避免。
2. **API 响应格式早确认**：应在第一次对接接口时就用 `console.log` 验证格式，而不是等到线上报错。
3. **字符串用双引号**：含缩写词的英文字符串一律用双引号，根绝弯引号问题。
4. **框架内置优先**：自定义 Plugin 引入了两个新 bug，而 `appType: 'spa'` 一行解决问题。
